## 1. 产品概述

### 1.1 产品愿景

**愿景陈述**: HypeTune 致力于成为新一代音乐爱好者的"品味放大器",通过 AI 技术将隐性的听感转化为显性的、高逼格的视觉与文字物料,帮助用户在流媒体时代重新确立"审美主权"。

> 基于 Ebi Atawodi 的洞察,我们的愿景必须平衡四个维度:足够振奋人心、切实可行、不受当前技术限制束缚、并植根于清晰的用户痛点。

### 1.2 问题定义

**核心问题**:
在流媒体时代,音乐变得廉价且易得,但用户表达品味的门槛反而变高了。大多数用户遇到好歌只会说"好听",缺乏足够高质量的社交货币来分享音乐体验。

**用户痛点**:
- **The Curator (分享狂)**: 每天高频分享音乐到朋友圈/Story,需要独特的文案和视觉来维持"懂行"人设
- **The Lucker (潜水员)**: 想发动态但词穷,需要"嘴替"来表达情感
- **The Digger (挖掘者)**: 对幕后制作名单(Credits)感兴趣,但懒得去查 Discogs 的硬核乐迷

> 遵循 Marily Nika 的建议,我们避免"为 AI 而 AI"的陷阱。AI 是手段,解决"表达品味门槛高"这一问题才是目的。

### 1.3 解决方案概述

HypeTune 是一款浏览器插件,它监听用户在主流音乐平台(Web 端)的播放行为,并通过 AI 生成两种核心资产:
1. **高逼格文案**: 基于歌曲特征生成三种风格的点评
2. **精美视觉卡片**: 包含封面、KwaKwa 形象、金句和元数据

**产品形态**: Chrome / Edge Extension (Manifest V3)

## 2. 核心功能需求

### 2.1 数据抓取层

#### 2.1.1 宿主平台支持
| 平台 | 支持状态 | 优先级 |
|------|----------|--------|
| 网易云音乐 | ✅ 支持 | P0 |
| QQ 音乐 | ✅ 支持 | P0 |
| YouTube Music | ✅ 支持 | P1 |

#### 2.1.2 抓取机制
**实现方式**: Content Script 监听 DOM 变化

**核心字段**:
- 歌名
- 歌手
- 专辑名

**兜底策略**:
若 DOM 结构变更导致抓取失败,允许用户手动输入歌名作为 Fallback。

### 2.2 夸夸生成引擎

#### 2.2.1 AI 模型
**基础模型**: Gemini 2.0 Flash Thinking

#### 2.2.2 三种生成模式

| 模式 | 用户意图 | 输出风格 | 数据源侧重 | KwaKwa 状态 |
|------|----------|----------|------------|-------------|
| **Emo (走心)** | 寻求共鸣,深夜分享 | 文学性、通感、场景化 | 歌词意象、调性 | Melancholy (垂头冠) |
| **Hype (上头)** | 宣泄情绪,安利神曲 | 短促有力、大量 Slang、感叹号 | BPM、能量值 | Hype (炸毛) |
| **Pro (懂行)** | 确立权威,展示品味 | 客观冷静、术语堆叠 | Discogs 数据 | Pro (黑框眼镜) |

#### 2.2.3 "烂歌防御"机制

**触发条件**: AI 判定歌曲具有"极简重复"、"低幼"或"土味"特征

**应对策略**: 功能主义评价。不夸艺术性,夸其"多巴胺释放效率"或"洗脑工程学成就"

**KwaKwa 表现**: Awkward (尴尬汗颜状态)

### 2.3 视觉生成器

#### 2.3.1 卡片组成
**视觉层级结构**:
1. **Header**: 歌曲封面 (调用 iTunes API 获取 600px+ 高清图)
2. **Character**: 对应状态的 3D KwaKwa 形象
3. **Content**: 核心金句(大字) + 辅助数据标签(如 Style: Synth-wave, Year: 1984)
4. **Footer**: 听歌二维码 + HypeTune 水印(Free 版)

#### 2.3.2 交互功能
**试听模块**: 在卡片生成页提供 iTunes 30s 试听按钮(自定义波形 UI),用于验证"高潮片段"

### 2.4 KwaKwa 交互系统

#### 2.4.1 KwaKwa 形象定义
**原型**: 玄凤鹦鹉 + 赛博格改造

**核心特征**:
- 喙部: 哑光金属扬声器网格
- 头冠: 动态音频波形软胶
- 配件: 半永久焊死的复古大耳机

#### 2.4.2 交互状态机

| 状态 | 行为描述 | 视觉表现 |
|------|----------|----------|
| **Idle (待机)** | 缩成球打瞌睡 | 鼻涕泡忽大忽小(呼吸感) |
| **Hype (激活)** | 睁大眼,疯狂点头 | 头冠炸开(锯齿波) |
| **Pro (专业)** | 神情严肃 | 戴黑框眼镜,穿高领毛衣,头冠变 EQ 柱状图 |
| **Overheat (限制)** | 吐黑烟 | 全身焦黑,头冠乱码,眼神 X_X |

## 3. 用户体验流程

### 3.1 标准用户旅程

```
潜伏阶段 → 触发唤醒 → 交互配置 → 生成交付 → 分享传播
```

**阶段详解**:

1. **潜伏**:
   - KwaKwa 以悬浮球形态栖息在页面右下角(Idle 状态)

2. **触发**:
   - 当歌曲切换时,KwaKwa 睁眼 2 秒,头顶冒气泡:"New Vibe Detected!"

3. **唤醒**:
   - 用户点击 KwaKwa,侧边栏 丝滑滑出

4. **配置**:
   - 插件自动读取歌名/歌手
   - 用户选择模式

5. **生成**:
   - 点击 "Hype It"
   - 后端并行处理: iTunes 抓图 + Discogs 查数据 + Gemini 写文案

6. **交付**:
   - 生成一张类似黑胶封套的精美卡片
   - 提供下载/复制功能

7. **限制**:
   - 若当日第 4 次点击,KwaKwa 触发 Overheat 状态,弹出付费引导

### 3.2 异常流程

| 异常场景 | 应对措施 |
|----------|----------|
| DOM 抓取失败 | 显示手动输入入口 |
| 网络请求超时 | 显示重试按钮,提示"网络拥挤,KwaKwa 在努力..." |
| AI 生成失败 | 返回降级文案:"这首歌的 Hype 正在加载中..." |
| 额度耗尽 | KwaKwa 触发 Overheat 状态,引导升级 |

## 5. 技术架构

### 5.1 整体架构

**架构理念**: 轻量化和快速迭代,采用 Serverless 架构

```
┌─────────────────────────────────────────────────────────┐
│                    Frontend (Extension)                 │
│  ┌─────────────┐  ┌──────────────────────────────────┐ │
│  │   React     │  │    Content Script (Injection)     │ │
│  │  + Vite     │  │  - DOM Observation               │ │
│  │  (Side Panel)│  │  - Data Extraction             │ │
│  └─────────────┘  └──────────────────────────────────┘ │
│         Manifest V3 + TailwindCSS                        │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│           Backend (API Aggregator - Cloudflare)        │
│  ┌─────────────┐  ┌──────────────────────────────────┐ │
│  │   Discogs   │  │         Gemini API               │ │
│  │     API     │  │   (Hype Engine + Safety Layer)   │ │
│  └─────────────┘  └──────────────────────────────────┘ │
│         iTunes Search API                              │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    Database (Supabase)                  │
│  • 用户额度管理  • Pro 状态  • 使用日志                  │
└─────────────────────────────────────────────────────────┘
```

### 5.2 技术栈清单

| 层级 | 技术选型 | 用途 |
|------|----------|------|
| **Frontend** | React + Vite | Side Panel UI |
| | TailwindCSS | 样式 |
| | Manifest V3 | Chrome Extension 规范 |
| **Backend** | Cloudflare Workers | 极低延迟,处理跨域请求 |
| **Database** | Supabase | 存储用户额度、Pro 状态 |
| **MVP 阶段兜底** | LocalStorage + 服务器时间戳校验 | 快速开发,允许极客破解 |

## 6. 成功指标

### 6.1 北极星指标

**核心指标**: **"Hype Completion Rate (HCR)"**
- 定义: 用户点击 "Hype It" 到成功生成卡片并完成分享/下载的转化率
- 重要性: 直接反映产品核心价值是否被用户认可

> 基于 Jess Lachs 的洞察,我们关注短期的可操作指标,而非仅关注长期留存这样的滞后指标。

### 6.2 关键指标

| 指标类别 | 具体指标 | 目标值(MVP 阶段) |
|----------|----------|-----------------|
| **增长** | Chrome 商店安装量 | 10,000+ (上线 30 天内) |
| | 日活跃用户 (DAU) | 1,000+ |
| **参与度** | 日均 Hype 次数 | 2.5 次/用户 |
| | 模式选择分布 | Emo 40% / Hype 35% / Pro 25% |

| | Pro 订阅数 | 250+ (上线 30 天内) |
| **质量** | 生成成功率 | > 95% |

### 6.3 阶段性目标

| 阶段 | 时间范围 | 核心目标 |

## 8. 风险评估与应对

| 风险类型 | 风险描述 | 应对措施 | 优先级 |
|----------|----------|----------|--------|
| **技术风险** | 宿主平台改版,DOM 抓取失效 | 1. 增加 DOM 嗅探鲁棒性<br>2. 提供手动输入 Fallback | P0 |
| | 高频请求导致 Discogs/Gemini 封 IP | 1. Cloudflare Workers 自带 IP 池<br>2. 增加服务端缓存(Redis) | P0 |
| **产品风险** | 用户不愿意安装浏览器插件 | 1. 通过社交传播建立信任<br>2. 强调"一键卸载"的轻量化特性 | P1 |
| | 生成内容质量不稳定 | 1. 烂歌防御机制<br>2. 用户反馈优化 Prompt | P0 |
| **内容安全** | AI 生成不当言论(涉黄涉政) | 1. Prompt 增加 Safety Settings<br>2. 关键词过滤层 | P0 |
| **商业风险** | 用户付费意愿不足 | 1. 3 vs 50 策略制造紧迫感<br>2. 精美皮肤作为付费钩子 | P1 |

## 9. 依赖与约束

### 9.1 外部依赖

| 依赖方 | 用途 | 约束 |
|--------|------|------|
| **Discogs API** | 获取专辑元数据、制作信息 | 免费版速率限制: 60 requests/min |
| **iTunes Search API** | 获取高清封面、试听链接 | 无公开速率限制 |
| **Gemini API** | 文案生成 | 成本敏感,需控制 token 使用量 |
| **Chrome Web Store** | 插件分发 | 审核周期 1-3 天 |
| **Cloudflare** | Serverless 托管 | Workers 免费版: 100,000 requests/day |

### 9.2 技术约束

- **浏览器兼容性**: 仅支持 Chrome/Edge (Manifest V3)
- **网络依赖**: 所有 API 调用需要网络连接
- **延迟要求**: 从点击 "Hype It" 到生成完成 < 3 秒

### 9.3 法律合规

- **数据隐私**: 不存储用户播放历史(仅存储额度数据)
- **版权合规**: 仅提供元数据和文案,不提供完整音频文件
- **用户协议**: 明确生成内容的 AI 属性
